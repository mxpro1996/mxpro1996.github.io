1. [VFS] dd命令的诡异现象

起因：

使用dd(conv=fdatasync)写入启动SD卡时，总是发现没有生效，以为dd命令存在缺陷。

解析：

当sd卡没有插入时,dd of=/dev/sda1会创建一个同名文件，这个文件会阻碍实际块设备分区文件的建立。

补充：

出于VFS的一切皆文件性质，当对设备操作无效时，考虑文件系统的限制规则，比如“同名文件”，一定要确定操作的是文件还是设备文件。

2. [GPT] root=PARTUUID=xxxxxxxxxxxxx

起因：

cmdline里对rootfs进行硬编码ID，由于sd卡启动的任意性，怀疑这个partUUID的来源

解析：

怀疑是正确的，ST官方确实硬编码了rootfs uuid，因此当分区表发生变化时，这种匹配是不可靠的，应该为root=/dev/mmcblk0p1这种形式，比起在卡烧录和emmc烧录时规定uuid要灵活很多。

补充：

deadline可以指定内核调度模式

5. [ST] display_controller apply config failed

起因：

6. [NFS] rootfs挂载与initrd互斥

起因：

在nfs启动rootfs时出现:

[    8.373114] Run /init as init process
Starting version 244.3+
root '/dev/nfs' doesn't exist or does not contain a /dev.

解析：

尝试NFS远程挂载rootfs时，不应该在UBOOT启动中追加INITRD，附带的精简版init进程无法精确找到nfs挂载的文件。

补充(cmdline)：

root=/dev/nfs rw nfsroot=192.168.10.20:/share/rootfs,tcp,vers=4 ip=192.168.10.21 rootwait earlyprintk

7. [ALSA] 开发板声卡无声，驱动和DTS正常

起因：

在内核完成声卡移植后，调用aplay播放音频无声。

解析：

当使用ALSA架构时候，PCM声源有两个来源（DSP/Direct），其中DSP是mixer混音工作的基础，如果不走dsp声音直出但是不可调，走DSP时必须先打开音频开关，否则静音。

补充：

路径选择看sget ‘DAC Mux’选项，可选dsp或者直出Direct。

numid=2,iface=MIXER,name='PCM Playback Switch'
  ; type=BOOLEAN,access=rw------,values=2
  : values=on,on
numid=1,iface=MIXER,name='PCM Playback Volume'
  ; type=INTEGER,access=rw---R--,values=2,min=0,max=127,step=0
  : values=54,54
  | dBscale-min=-51.50dB,step=0.50dB,mute=0
amixer cset name='PCM Playback Switch' 'on','on'
# 开启dsp-PCM
amixer cset name='PCM Playback Volume' '63','63'
# 设定音量
amixer cset name='Analog Playback Volume' '204','204'
amixer cset name='PCM channel mixer' 'L R'

8. [BT&Wlan] AP6212 BCM网卡/蓝牙模块加载

起因：

单独内核编译的内建驱动，无法唤起网卡工作

解析：

该网卡的工作依赖固件加载，而加载的逻辑整合在brcmctl等等模块中，虽然也有用户态手动加载固件的官方工具，但不如直接触发modprobe加载/lib/modules目录直接，模块则直接访问/lib/firmware读取固件。

补充：

用到的包：Bluez(HCI面向串口蓝牙)，Wpa_supplicant

[root@buildroot]:~# wpa_passphrase "你的 WiFi SSID" > wifi.conf
[root@buildroot]:~# wpa_supplicant -B -c wifi.conf -i wlan0
[root@buildroot]:~# udhcpc -i wlan0 -b # 必须广播DHCP形式，否则死机
[root@buildroot]:~# echo "nameserver 114.114.114.114" > /etc/resolv.conf
[root@buildroot]:~# echo "nameserver 8.8.8.8" >> /etc/resolv.conf
~# ping 360.com

# Or
# Prepare WPA Structure at conf
wpa_supplicant -Dnl80211 -iwlan0 -c/etc/wpa_supplicant.conf
# WPA_CLI(optional)
wpa_cli -i wlan0

9. 在Win32下交叉编译QT-6.6.3失败

起因：

缺乏合适可用的工具链，Linaro-7.5.0过旧（不具有charconv），ARM-9.2 10.2 11.2这些连bits/types/__FILE目录都无法定位，头文件包含存在严重问题。

暂时解法：

降级QT-5.14.2配合GCC-7.5.0进行win32下依赖库预编译，出自制工具链外暂无其他良解。

10. git-windows突然不再读取密钥

起因：

安装Cadence-Allegro会破坏用户的HOME环境变量，指向"C:\Users\Username\SPB_Data"

解法：

删除多余的SPB_Data这一级目录，仍然不行就在.ssh的config文件中追加：

Host github.com
Hostname ssh.github.com
Port 443
IdentityFile {path of your rsa-key}

10. uboot不启动boot分区

起因：

在pxeboot无盘启动时，会修改pxelinux.cfg为目录名，mac地址为设备配置文件名。转换成烧录镜像时候必须修改配置文件布置。

解法：

在uboot本地启动时，会执行boot.scr.uimg里的脚本，官方采用这个脚本指定{boardname}_extlinux.conf，对于fsmp1a来说boardname=‘stm32mp157a-fsmp1a’。但我们暂时不需要自定义，采用默认mmc0/1_extlinux路径extlinux.conf引导配置即可。

15. [QT] Q_PROPERTY对变量的两种联系

解法：

  1. READ/WRITE指定访问方法
  2. MEMBER声明直接关联变量 (自动发送信号)
    其中1定位尴尬，不指定WRITE时无法访问变量，指定时系统无法重写该方法，只能手动发射。
    2因为直接由系统生成Get/Set，所以可以搭配NOTIFY实现改变时的信号发射
    且必须使用SetProperty/Property来代行访问，才能产生信号。
网上查阅发现这么一个说法：这里仅是告诉元对象系统 mask 属性改变时会发出 maskChanged 信号，但是它不是实现，具体的实现需要我们自己来完成，我们还是需要在类内相关方法下添加 emit 进行信号发送，Qt这个设计很迷。
如：
void MyPropertyClass::setMask(QString strMaskNum)
{
    m_mask = strMaskNum;
    emit maskChangedXH(m_mask);
}
也就是说，以下两段代码起到的效果是完全一样的，不管你加不加，这个信号都需要你自己 emit ：
Q_PROPERTY(QString mask READ getmask WRITE setMask NOTIFY maskChanged)        //有 NOTIFY maskChanged
Q_PROPERTY(QString mask READ getmask WRITE setMask)    　　　　　　　　　　　　　　//没有 NOTIFY maskChanged

例外：
但以下方法这个信号是会自动发出的，不用你添加相关的 emit 信号也会发出：
Q_PROPERTY(QString mask MEMBER m_mask NOTIFY maskChanged)
使用 MEMBER 注册后，当 mask 发生改变，将会发出 maskChanged 信号
MEMBER含义为：如果一个属性不需要 READ ，但又想使用属性系统，可以使用MEMBER来注册，MEMBER后面是成员变量的名字；

16. [QT] 基于metaCall

解法：

signal声明实际上对应moc代码：

// SIGNAL 0
void MXObject::ageChanged(int _t1)
{
    void *_a[] = { nullptr, const_cast<void*>(reinterpret_cast<const void*>(std::addressof(_t1))) };
    QMetaObject::activate(this, &staticMetaObject, 0, _a);
}
HashMap(SLOT,SIGNAL) connectionList 连接关系表
所谓SLOT和signal的匹配，其实由元数据查找sender的signal id，将可能附带的形参一起带给receiver的SLOT函数执行，也即是metacall(SLOT())。

17. [Flash] UBI-FS文件系统流程

原理：

 ubi是一种flash文件系统，基于mtd（Memory Technology Device)，mtd负责实际管理nor/nand flash芯片时序，向fs文件上层抽象出mtd0 mtdblock0，字符设备和块设备，其中块设备不含ioctl。

解法：

# 首先，在cmdline追加mtd分区情况
# 其次，格式化mtd3分区
ubiformat -y -d 3
# 然后，映射mtd3到ubi3文件设备
ubiattach /dev/ubi_ctrl -m 3 -d 3
# 在ubi3上新建卷(Volume 0，ubi3_0)
ubimkvol -m 3 -n 0
# (可选）ubirmvol可以删除卷
# 挂载ubi3_0
mount -t ubifs /dev/ubi3_0 /mnt/nor
